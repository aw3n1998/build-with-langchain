## 二、核心项目介绍（AgentLab）

### 用业务语言说项目

**项目背景：**
自主设计并从零实现了一套 AI 多 Agent 协作系统，模拟企业级 AI 自动化工作流。

**解决的业务问题：**
单一 AI 无法同时处理多类任务（写代码、读文件、问答），且串行处理效率低。

**我设计的方案：**

```
用户输入
   ↓
意图识别层（主调度 Agent）
   ↓  根据意图并行分发
代码执行 Agent ── 文件处理 Agent ── 通用问答 Agent
   ↓  全部完成后汇聚
结果整合层（统一输出）
```

**关键设计决策（面试时可展开）：**

| 设计点 | 我的选择 | 业务价值 |
|--------|---------|---------|
| 并行执行 vs 串行 | 并行（同时触发多个 Agent） | 响应速度提升，复合型任务不再等待 |
| 工具隔离 | 每个 Agent 只看到自己需要的工具 | 防止 AI 乱用工具，降低出错率 |
| 工具动态检索 | 用语义相似度按需加载 Top-K 工具 | 工具再多也不浪费 token，可无限扩展 |
| 自动纠错 | 工具报错直接喂回 AI，自动修正 | 减少人工干预，提高自动化率 |
| 批量任务并行 | 动态创建 N 个 Worker 同时处理 | 100 个任务不串行等待，整体耗时等于最慢单个 |
| 会话持久化 | 内置 SQLite，按会话 ID 隔离 | 支持多用户并发，重启不丢上下文 |
| 可视化调试 | 接入 LangGraph Studio | 流程节点可视化，便于非技术同学理解 |

而且我有一个额外的优势：我理解开发的实现成本和技术边界，能设计出真正能落地的方案，而不是停留在流程图上。"

---

### Q2：你了解 AI Agent 和 Skill 是什么吗？

"我不只是了解，我亲手实现过。

Agent 本质是一个能感知输入、调用工具、输出结果的自主执行单元。Skill 是 Agent 能调用的能力模块，比如读文件、执行代码、查询数据库。

在我的项目里，我设计了代码执行、文件读取、实时查询三类 Skill，分别注入不同的专业 Agent，避免 Agent 因为工具太多而产生决策混乱。

更进一步，我还实现了 Skill 的动态检索机制：把所有工具的描述做成向量索引，用户每次发消息时语义搜索最相关的几个工具注入给 AI，其余工具完全不占上下文——这样 Skill 数量可以无限扩展，不会影响 AI 的判断质量。

从业务角度说，设计 Skill 的核心原则是：**最小权限 + 单一职责**，这样才能保证 Agent 行为可预期、可审计。"

---

### Q3：你没用过 Coze 或 Dify，怎么快速上手？

"我没有直接用过这些平台，但我理解它们底层的运行逻辑——因为我是从底层手写过来的。

Coze 和 Dify 本质上是把 Agent 编排、工具调用、流程路由做成了可视化界面，我用 LangGraph 手写实现了相同的能力。

上手这类平台对我来说主要是熟悉 UI 操作，底层逻辑我已经清楚。给我一天时间，能搭出基础流程；给我一周，能设计复杂的多分支工作流。"

---

### Q4：你怎么判断哪些环节适合 AI 做，哪些需要人工？

"我的判断框架是三个维度：

**规则明确度**：如果一个任务能写成清晰的 if-else 判断树，AI 可以完全接管。如果需要大量上下文理解和价值判断，需要人工介入。

**容错成本**：出错了影响有多大？审核金融合规文件和回复用户闲聊，容错成本完全不同，后者可以高度自动化，前者需要 AI 初判 + 人工复核。

**数据可得性**：AI 需要输入才能输出。如果关键信息散落在各处无法结构化，先做数据治理，再谈 AI 自动化。

在我的项目里，代码执行结果自动纠错，但我在设计文件写入权限时做了限制——因为写操作不可逆，这类高风险操作需要保留人工确认节点。"

---

### Q5：描述一下你会怎么设计一个 AI 自动化流程？

"我的设计步骤是：

1. **梳理现有流程**：把人工操作步骤画出来，找出重复性高、规则明确的环节
2. **识别痛点**：哪个环节最慢？哪个最容易出错？哪个人力成本最高？
3. **拆解 AI 节点**：确定哪些步骤交给 AI，哪些保留人工，设计清楚每个节点的输入、输出、异常分支
4. **定义兜底机制**：AI 置信度低时怎么办？超时怎么办？错误怎么告警？
5. **小规模验证**：先在一个子场景跑通，收集数据，再推广

以客服场景举例：意图分类 → AI 处理（退换货查询、订单状态）→ 复杂问题转人工 → 人工处理结果反哺 AI 训练数据。"

## 四、可能的深度追问

**"你的 AgentLab 项目解决了什么实际业务问题？"**

> "目前是学习性质的技术项目，验证了多 Agent 协作架构的可行性。但它的架构模式可以直接映射到真实业务：主调度器对应业务路由层，专业 Agent 对应各职能系统，Aggregator 对应结果汇总和报告生成。"

**"如果 Skill 非常多，系统怎么处理？"**

> "这是我专门解决过的问题。我用向量数据库（FAISS）存储所有 Skill 的语义描述，用户每次输入时做相似度检索，只把最相关的 Top-K 个 Skill 传给 LLM，其余完全不加载。这样 Skill 数量从 4 个扩展到 4000 个，对 LLM 的 token 消耗没有任何影响。"

**"批量任务怎么处理，比如同时生成大量内容？"**

> "我实现过 Map-Reduce 模式：由规划节点把大任务拆分成 N 个子任务，动态创建 N 个 Worker Agent 并行执行，全部完成后由汇聚节点统一整理结果。Worker 数量由 AI 运行时决定，不需要硬编码。100 个任务的耗时等于最慢单个任务，而不是 100 倍。"

**"你能输出需求文档和流程图吗？"**

> "可以。我做开发时需要先看懂需求文档，也写过技术方案文档。流程图我习惯用 Mermaid 或者画板工具，能输出泳道图、状态机图、数据流图。"

## 七、行业专项：成人内容平台场景

> 该行业本质是内容平台，AI 流程设计核心问题与视频网站、社交平台一致，额外多合规和安全要求。

### 高频场景问题 · 准备答案

---

**Q：怎么设计内容上传后的自动审核流程？**

"我会设计一套 AI + 人工的双层审核流程：

第一层 AI 初筛：对上传内容做违禁检测、合规标签识别，按置信度分三类——高置信度合格自动通过、高置信度违规自动拒绝、灰区内容流转到人工审核队列。

第二层人工节点：AI 拿不准的交给审核员，审核结果同时反哺 AI 模型，持续提升准确率。

关键设计原则是：绝不让 AI 单独处理高风险判断，必须有人工兜底节点，出错成本太高。"

---

**Q：几百万视频怎么自动打标签？**

"核心是用语义向量把视频内容映射到标签空间。具体流程：分析视频封面、标题、描述 → AI 生成候选标签 → 与已有标签体系做语义对齐 → 多语言自动翻译。

这和我做过的 Skill 动态检索原理一样——用 Embedding 做语义匹配，不需要人工一条条对应。"

---

**Q：创作者从上传到发布要经过哪些 AI 自动化节点？**

"我会设计这样一条流水线：

```
上传 → 自动截取最优封面 → AI 生成标题/描述建议
     → 自动多语言翻译 → 审核通过 → 发布+推荐系统同步
```

各节点并行执行，不串行等待——审核、打标签、转码可以同时跑，整体速度是最慢单个节点的时间，而不是所有节点加总。"

---

**Q：不同国家法规不同，怎么设计合规流程？**

"合规规则不能硬编码进流程，要抽象成可配置的策略层。用户地区 → 匹配对应规则集 → 决定内容是否展示、是否需要额外验证。这样新增地区的法规只需要加一条策略，不需要改流程本身。"

## 八、面试经验（复盘）

**最重要的一条：技术是底气，但要用业务语言说出来。**

三步答题节奏：
1. **先说场景**（"当工具数量很多时..."）
2. **再说问题**（"全部加载会浪费 token 且干扰注意力"）
3. **最后说方案**（"我用语义检索动态筛选"）

**绝对不要说的话：**
- "我用 Send API 做了 fan_out" → 改成 "我实现了动态任务分发，Worker 数量运行时决定"
- "operator.add reducer" → 改成 "所有并行结果自动汇聚，不会互相覆盖"
- "FAISS IndexFlatIP" → 改成 "向量相似度检索，找最相关的工具"

**你的真实竞争优势：**
你不只是"用过 AI 工具"，你理解 AI 系统的内部运作——这让你能在设计阶段判断什么方案能落地、什么方案是空中楼阁。这是纯业务背景候选人没有的东西。
